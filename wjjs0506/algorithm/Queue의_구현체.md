# Queue의 구현체

Java로 알고리즘 문제를 풀다보면 `BFS(Breadth-First Search)` 등에서 Queue의 구현체로 LinkedList를 사용한다.

`Queue<T[]> q = new LinkedList<>();`

왜 Queue 자료형의 구현체로 주로 LinkedList를 사용할까?

## 알고리즘에서 Queue를 사용할 때

알고리즘에서 Queue를 사용할 때는 대부분 **"FIFO(First-In, First-Out)"** 구조를 사용해야 할 때이다.

이는 대부분의 데이터 수정이 맨 앞을 삭제하거나 맨 뒤에 추가하는 작업일 것을 뜻한다.

### ArrayList의 경우

ArrayList로 Queue를 구현했을 때, 데이터를 삭제(poll)를 한다고 하면, 0번째 인덱스가 사라지고 1~N번째 인덱스의 모든 데이터를 한 칸씩 앞으로 당겨야한다.
(ArrayList는 번호가 매겨진 연속된 메모리 공간을 사용하므로)

그러므로 삭제 작업의 소요 시간은 O(N)이다. 이는 데이터가 많을 수록 오래걸린다.

### LinkedList의 경우

반면, LinkedList로 Queue를 구현했을 때, 데이터를 삭제하는 데는 O(1)의 시간이 소요된다.  
LinkedList는 `[데이터 | 이전,다음 노드 주소]`의 형식으로 따로 떨어져 있고 Node들이 고리로 연결된 형태이기에

```
1. 맨 앞 칸(Head) 제거
2. 두 번째 Node를 새로운 (Head)로 지정
```

이렇게만 하고 뒤에 연결된 데이터들은 건들지 않아도 되기 때문이다.

### 그럼 항상 LinkedList가 우수할까??

꼭 그렇지만도 않다. 두 자료형의 특성 상 `데이터 조회(GET)`에서는 ArrayList의 성능이 더 우수하기 때문이다.

[206번째 데이터에 접근할 떄]

1. ArrayList : 데이터가 연속되기에 시작점에서 206번째의 데이터에 접근
2. LinkedList : 1번째 데이터(2번지) 접근 -> 2번째 데이터(501번지) 접근 -> 3번째 데이터(209번지) 접근 -> ... -> 206번째 데이터(43번지) 접근

이처럼 오히려 데이터를 조회할 때는, ArrayList가 O(1)인 상수시간이 소요되고, LinkedList는 O(N)의 시간이 소요된다.

그렇기에 작업이 어떤 성격(GET을 많이하는 지, Insert/Delete를 많이 하는 지)를 파악하고 선택하면 된다.

## 그래서 Queue는 무엇으로 하면 되나요??

재미있는 건 우린 이미 답을 알고있다.  
**Queue의 설계 원칙**을 고려해보면 해당 질문에 대한 답을 쉽게 할 수 있다.

이전 질문인 알고리즘에서 Queue사용할 때는 **"FIFO(First-In, First-Out)"** 구조를 사용해야 할 때이기에 `get(index)`를 자주 하지 않는 구조이다.

만약 중간에 있는 데이터를 자꾸 조회애햐 하는 요구사항의 경우, Queue가 아니라 **"일반적인 List"** 가 필요한 경우이므로 Queue를 사용하지 않아도 된다.

즉, 거의 모든 경우 LinkedList로 구현하면 된다.

## 번외) ArrayDeque

요즘은 LinkedList보다 ArrayDeque를 더 선호한다.

ArrayDeque(Array Double Ended Queue)는 Array 배열 기반이지만 이름과 같이 양쪽에서 데이터를 추가하고 제거할 수 있는 원형구조를 형성하여 양쪽 앞/뒤 삽입 삭제가 O(1)이고, LinkedList의 다음 노드 및 이전 노드에 대한 정보를 저장하고 있지 않아도 되어 메모리적으로도 효율적이다(= 메모리 오버헤드가 발생하지 않는다).

1. 설계의 목적 : Deque 설계 시 Queue와 같은 목적으로 설계 양 끝에서만 데이터를 넣고 뺀다는 규칙을 강제 (중간에서 꺼낼 수 있게 된다면 그것은 그냥 List) => ArrayDeque는 `get(int index)` 메서드를 제공하지 않는다.
