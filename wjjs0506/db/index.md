# Index

추가적인 저장 공간을 활용하여 데이터베이스 테이블의 검색 속돌르 향상시키기 위한 자료구조

인덱스를 활용하면 SELECT 외에도 UPDATE나 DELETE의 성능도 같이 향상된다 -> 해당 연산의 대상을 찾기 위해 SELECT를 하기 때문!

## 장점

- 테이블 조회하는 속도와 그에 따른 성능 향상

## 단점
- 인덱스를 관리하기 위한 저장 공간 필요
- 잘못 사용할 경우 오히려 성능이 저하 -> 규모가 작은 테이블의 경우, 데이터 중복도가 높은 경우 etc

## 인덱스의 자료 구조

### 해시 테이블(Hash Table)

해시 자료구조를 사용해서 (key, value)형식으로 데이터를 저장하는 자료구조  
인덱스 값이 데이터 값에 대한 키로 동작하는 Hash Table(Bucket)을 만들어 대상을 바로 조회해 O(1)의 시간복잡도를 가지며 매우 빠른 검색을 지원한다.

하지만, 하나의 대상이 아닌 범위 (**부등호 연산** >, <)이 자주 사용되는 실서비스의 메인 데이터베이스 검색으로는 적합하지 않다.

(MySQL InnoDB등에선 B-Tree로 조회하다 자주 조회되는 값을 Hash 메모리에 등록하는 Adaptive Hash Index도 사용한다. -> 대부분 캐시성으로 사용)

**장점**
- 하나의 데이터를 찾는 데 매우 빠른 검색을 지원한다 (Redis와 같은 Key-Value Store)
- 고유한 식별자를 가진 테이블에 매우 적합하다

**단점**
- 범위 검색에 부적합하다

### B-Tree

이진탐색트리를 2개 이상의 자식 노드를 가질 수 있도록 확장한 자료구조

자녀 노드의 최대 개수를 원하는대로 결정해서 사용할 수 있다.
이진탐색트리를 확장한 구조이므로 O(log n)의 시간복잡도를 가지며 키 값에 따라 정렬된 상태로 데이터가 저장된다.

## 데이터 저장 전략

B-Tree 인덱스를 이용해서 주로 아래와 같은 전략으로 데이터를 저장한다.

#### Clustered Index

테이블 당 **1개**의 컬럼으로 물리적으로 정렬된다. (주로 PK)  
Non-Clustered Index보다 실제 데이터가 정렬되어 있어 속도면에서 우수한 성능을 보이지만, 데이터 변경(추가/수정/삭제) 시 매번 정렬을 수행해야 하기 때문에 추가/수정/삭제 시 성능은 떨어진다.

#### Non-Clustered Index

책의 찾아보기와 같이 따로 데이터의 주소(포인터)가 들어간 데이터 페이지를 구성 (주로 Unique 제약조건)  
테이블 당 여러개 존재 가능하며, 리프 노드에 실제 데이터가 아닌 주소가 있으므로 실제 데이터를 찾기 위해 한 번 더 접근해야하는 RID Lookup or Key Lookup 이 필요하다.
