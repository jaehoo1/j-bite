# Infrastructure 폴더

프로젝트 컨벤션 중 docker-compose.yml 파일을 infrastructure 패키지 하위에 두기로 하여 infrastructure 폴더에 대해 알아보기로 하였다.

패키지 구조는 코드의 가독성, 유지보수성, 확장성을 결정짓는 중요한 요소이므로 시작 전 고민하는 것이 좋다

## 계층형 구조 vs 도메인형 구조

### 계층형 구조 (Layered Architecture)

역할(기능적 계층)에 따라 패키지를 나누는 방식

**선택 기준**
- 프로젝트 규모: 소규모 프로젝트, 프로토타입을 개발할 때 적합
- 팀의 숙련도: MVC 패턴에 익숙한 경우, 빠르게 협업가능
- 단순한 비즈니스 로직: 로직이 단순하고, 서비스 간의 의존성이 낮을 때 유리

### 도메인형 구조 (Domain-Driven Architecture)

비즈니스 단위(도메인)에 따라 패키지를 나누는 방식

**선택 기준**
- 프로젝트 규모: 중대형 프로젝트, 비즈니스 요구사항이 복잡한 경우 적합
- 확장성과 유지보수: 장기적 기능 추가(유지보수성), 각자 다른 도메인을 전담해야 할 때 적합
- 도메인 중심 설계(DDD): 도메인 지식이 핵심인 프로젝트라면, 코드 구조 자체가 비즈니스 언어를 반영하도록 설계

infrastructure 패키지의 경우 주로 DDD(도메인 주도 설계)의 4계층에서 사용된다.

## Infrastructure 패키지

데이터베이스, 외부 API, 메시징 시스템, 파일 시스템 등과 같은 기술적인 세부 사항을 담당하는 계층

### 사용 시 장점

1. 유지보수성과 확장성 향상
    - 데이터베이스, 메시징 시스템, API 연동 방식이 변경되더라도 Infrastructure 계층에서만 수정하면 되어, 전체적인 코드 변경이 최소화된다.
2. 비즈니스 로직과 기술적 관심사 분리 (관심사의 분리 가능)
    - 메인 계층은 비즈니스 로직을 담당
    - Infrastructure 계층은 기술적인 구현을 담당
3. 다양한 인터페이스와의 연동 가능
    - 위 두 기능을 통한 특정 기술에 종속되지 않고 다양한 인터페이스 사용 가능
    - 인터페이스 변경 시 수정 용이

### 주로 두는 관심사

1. 데이터베이스
    - noSQL
    - Mysql
2. 외부 API 통신들
    - docker
    - 외부 modules
3. 메세징 시스템
    - KAFKA
    - RabbitMQ
4. 로깅
    - log4j
    - logback
5. 보안
    - JWT
    - Spring Security

### DDD 4계층에서의 infrastructure 패키지

비즈니스 로직(Domain)이 특정 기술(JPA, AWS, Redis 등)에 오염되지 않도록 격리하는 벽의 역할을 한다.

**의존성 역전(DIP)의 활용**
- Domain 계층: "데이터 저장"기능 수행을 위한 Repository 인터페이스만 정의 (기술이 무엇인지 모름)
- Infrastructure 계층: 구현체(MySQL, MongoDB)를 결정하고 위 인터페이스를 실제로 구현

### DDD 4계층 외의 intrastructure 패키지

1. 기술 교체 가능성이 있을 때
    - 로컬 파일 시스템 -> AWS S3 진행, 인터페이스는 상위 계층에 두고 구현체만 infrastructure에 둠
2. 외부 라이브러리 의존성을 격리하고 싶을 때
    - 프로젝트 전반에 특정 외부 라이브러리 코드가 퍼지는 것을 막고 싶을 때 사용
3. 코드의 '성격'을 명확히 하고 싶을 때
    - 비즈니스 규칙인가(Domain/Service)과 단순한 도구(Infrastructure)를 구분하여 가독성 향상
